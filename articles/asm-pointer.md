## なぜポインタという記法が存在するのか

レジスタに `0x2000` という数値が入っている。

この `0x2000` は「メモリ上のアドレス」なのか、「ただの数値」なのか——レジスタに入った状態では、区別できない。

```
r3 = 0x2000   ← これはアドレス？ それともただの数値？
```

CPUはレジスタの中身だけを見ても判断できない。判断するのは**命令**だ。

```
ldr r0, [r3]      ; r3 をアドレスとして扱い、r3 が示すメモリ番地の値を読む
add r3, r3, #1    ; r3 をただの数値として扱い、+1 する
```

同じ `r3 = 0x2000` でも、使う命令によって「アドレス」にも「数値」にもなる。

だからプログラマーは**「この値はアドレスとして使う」という意図をコンパイラに伝える必要がある**。C言語でそれを伝える手段が、ポインタ型（`*`）という記法だ。

```c
int  x = 0x2000;   // ただの数値として扱え → add/mov 命令を生成
int *p = 0x2000;   // アドレスとして扱え  → ldr/str 命令を生成
```

ポインタは「難しい概念」ではなく、**「この値をアドレスとして使え」とコンパイラに伝えるための印**だ。

## ポインタが本当に必要になる場面

よく教科書に出てくるこのコードは、正直あまり実用的ではない。

```c
int x = 10;
int *p = &x;
*p = 20;   // これは x = 20 と同じ
```

ポインタが本当に必要になるのは、**C言語の `return` が1つの値しか返せない**という制約にぶつかったときだ。

Python を使ったことがある人なら、こんなコードを書いたことがあるはずだ。

```python
def min_max(a, b):
    return min(a, b), max(a, b)  # 2つ同時に返せる
```

C言語にはこれができない。`return` で返せるのは1つだけだ。

では「2つの変数を入れ替える」関数を書きたいとき、どうするか。

```c
void swap(int *a, int *b) {
    int tmp = *a;
    *a = *b;
    *b = tmp;
}

int main(void) {
    int x = 3, y = 7;
    swap(&x, &y);  // x=7, y=3 になる
}
```

`swap` は `return` では絶対に実現できない。2つの変数を同時に書き換える必要があるからだ。ポインタで両方のアドレスを渡すことで、関数の内側から2つの変数を直接書き換えられる。

**「return で返せない結果を関数の外に伝える」——これがポインタを使う必然性だ。**

## アセンブラで見ると「ただのアドレス計算」になる

`x=3, y=7` で `swap(&x, &y)` を呼び出したとき、アセンブラで見てみよう。

ARM では関数の引数を r0〜r3 のレジスタで渡す決まりがある。第1引数が r0、第2引数が r1 だ。

`main` の中でのスタックレイアウトはこうなっている。

```
高アドレス
  [ 保存した LR  ]
  [ 保存した r7  ]
  [ x = 3       ]  ← [r7, #8]
  [ y = 7       ]  ← [r7, #4]
低アドレス          ← SP / r7 はここ
```

r7 を基準に、高アドレス方向へ変数が並んでいる。`[r7, #8]` が x、`[r7, #4]` が y のアドレスだ。

まず **main の中**で、x と y のアドレスを計算して swap に渡す。

```
【main の中】
add  r3, r7, #8    ; main の x のアドレスを計算
mov  r0, r3        ; x のアドレスを第1引数（r0）に入れる
add  r3, r7, #4    ; main の y のアドレスを計算
mov  r1, r3        ; y のアドレスを第2引数（r1）に入れる
bl   swap          ; swap を呼び出す
```

`add r3, r7, #8` ——これが `&x` の正体だ。スタックポインタ（r7）からのオフセットで**番地**を計算して、レジスタに入れているだけ。「アドレスを渡す」とは、**ただの数値（アドレス番地）を渡しているだけ**だった。

次に **swap の中**では、受け取ったアドレス（r0/r1）を使って main のスタック上の変数を直接読み書きする。

```
【swap の中】
    ldr  r2, [r0]      ; r0 が指し示すメモリ番地の値を読み出す（tmp = *a ← main の x）
    ldr  r3, [r1]      ; r1 が指し示すメモリ番地の値を読み出す（*b ← main の y）
    str  r3, [r0]      ; r0 が指し示すメモリ番地に値を書き込む（*a = *b）
    str  r2, [r1]      ; r1 が指し示すメモリ番地に値を書き込む（*b = tmp）
    bx   lr
```

swap は「r0/r1 に入っているアドレスのメモリを操作する」だけで、それが main のスタック上の変数だとは知らない。アドレスを受け取って、そこを読み書きするだけだ。

**これがポインタの本質だ。アドレスを渡すことで、関数をまたいで同じメモリを操作できる。**

`ldr r2, [r0]` ——r0 が指し示すメモリ番地の値を読み出す。これが `*a` の正体。

`str r3, [r0]` ——r0 が指し示すメモリ番地に値を書き込む。これが `*a = *b` の正体。

**間接参照（`*`）とは、「レジスタの値をアドレスとして使え」という命令だった。**

## 配列とポインタはアセンブラで区別がつかない

```c
int a[3] = {1, 2, 3};
a[1] = 99;
```

アセンブラでは：

```
add  r3, r7, #8      ; a の先頭アドレスを計算
str  r2, [r3, #4]    ; [r3 + 4バイト] = 99 → a[1] への書き込み
```

`[r3, #4]` は「r3 のアドレスから4バイト先」という意味だ。これが `a[1]`（= 先頭から1要素 × 4バイト先）の正体。

**配列のインデックスアクセスは、アドレスのオフセット計算だった。**

C言語が「配列」と「ポインタ」を別の概念として教えているのは人間のための抽象化であって、アセンブラレベルでは同じことをしている。

## まとめ

| 疑問 | 答え |
|---|---|
| なぜポインタが必要か | レジスタは「アドレスか値か」を区別しない。プログラマーがコンパイラに伝える必要がある |
| `&x` とは何か | x が置かれているスタック上のアドレスを計算した数値 |
| `*p` とは何か | p に入っている数値をアドレスとして使い、そこからメモリを読み書きすること |
| 配列と何が違うか | アセンブラレベルでは同じ。オフセット計算をしているだけ |

ポインタは難しくない。**「この数値をアドレスとして使え」という指示をコンパイラに伝える記法**だ。
