1. What is the Stack?

The stack is a Last In, First Out (LIFO) data structure. Think of stacking books — you take the top book off first.

Push: place A, then B, then C on top
Pop:  remove C first, then B, then A

During program execution, this LIFO structure is used to temporarily hold local variables, return addresses, and saved registers.

2. Where in Memory is the Stack?

A program's memory is roughly divided into these regions:

High address ↑
┌─────────────────────────┐
│  Stack                   │  ← local variables, return addresses, etc.
│   ↓ (grows downward)     │
│                          │
│  Heap                    │  ← dynamic allocation (malloc)
│   ↑ (grows upward)       │
│                          │
│  Data segment (.data)    │  ← global/static variables
│  Code segment (.text)    │  ← instructions
└─────────────────────────┘
Low address ↓

The key point: the stack grows from high addresses toward low addresses. Each time you push a value, the address gets smaller.

Note
"The stack grows downward" means addresses decrease as the stack fills. Don't confuse this with the visual direction on screen (diagrams usually draw high addresses at the top).
Try it in AsmWalker
The Stack panel displays high addresses at the top. Each PUSH adds a new cell at the bottom — that is the stack growing toward lower addresses.

3. What PUSH and POP Really Do

PUSH and POP look simple but internally perform two steps: moving SP + reading/writing memory.

ARM Cortex-M

; PUSH {r4} expands to:
sub sp, sp, #4    ; SP -= 4
str r4, [sp]      ; [SP] ← r4

; POP {r4} expands to:
ldr r4, [sp]      ; r4 ← [SP]
add sp, sp, #4    ; SP += 4

x86-64

; push rbp expands to:
sub rsp, 8        ; RSP -= 8
mov [rsp], rbp    ; [RSP] ← rbp

; pop rbp expands to:
mov rbp, [rsp]    ; rbp ← [RSP]
add rsp, 8        ; RSP += 8

ARM registers are 4 bytes (32-bit), x86-64 registers are 8 bytes (64-bit), so the SP adjustment amounts differ.

Try it in AsmWalker
The Instruction Detail panel describes a PUSH as rsp ← rsp - 8; [rsp] ← rbp — showing the two-step SP-move + memory-write as a single instruction.

5. Why Functions Use the Stack

Functions need the stack for three main purposes:

  1. Saving local variables — temporary storage for variables declared inside the function
  2. Saving the return address — where to return after the function finishes (x86: call pushes it automatically)
  3. Saving registers (callee-save) — preserving the caller's register values so the callee doesn't clobber them

Together, these make up the stack frame — the section of stack dedicated to one function call. A new frame is pushed when a function is called and released when it returns.

Stack layout for main → FuncA → FuncB

With three levels of nested calls, the stack holds three frames stacked up. main's frame (called first) sits at the highest address; FuncB's frame (currently executing) sits at the lowest.

"Stack = push/pop only" is a misconception
Many beginners think that stack memory can only be accessed via PUSH and POP. In reality, two separate rules apply:
  • Between frames: frames are stacked and released in LIFO order
  • Within a frame: data is accessed by ordinary memory reads/writes (random access is fine)

Local variables are accessed freely by offset from the frame pointer: [fp - 4], [fp - 8], etc. PUSH/POP is only used for frame setup/teardown and saving/restoring registers.

6. Allocating and Releasing Local Variables

Space for local variables is reserved simply by moving SP. No system call like malloc is needed — this is very fast.

ARM — allocating locals

push {fp, lr}       ; save FP and LR
add  fp, sp, #4     ; set frame pointer
sub  sp, sp, #8     ; reserve 8 bytes for locals

; ... function body ...

sub  sp, fp, #4     ; restore SP (release locals)
pop  {fp, pc}       ; restore FP, PC = return address

x86-64 — allocating locals

push rbp             ; save RBP
mov  rbp, rsp        ; set frame pointer
sub  rsp, 16         ; reserve 16 bytes for locals

; ... function body ...

leave                ; mov rsp, rbp; pop rbp
ret                  ; jump to return address

"Releasing" just means moving SP back

Releasing stack space means moving SP back to where it was. The memory is not cleared.

Allocate: sub rsp, 16  ; RSP decreases → that region is "in use"
Release:  add rsp, 16  ; RSP increases → that region is "available" again
                       ; ※ data in memory remains until overwritten
Dangling pointer warning
Never return the address of a local variable from a function. Once the function returns and SP is restored, that memory becomes "free". The next function call will overwrite it, and your returned pointer will point to garbage (undefined behavior). 
int* bad() {
    int x = 42;
    return &x;  // danger: garbage value after function returns
}

Accessing local variables

Variables in the allocated region are accessed via the frame pointer (FP / RBP) plus an offset.

; Storing local variable x = 3 on the stack (x86-64)
mov  DWORD PTR [rbp-4], 3   ; x is placed at rbp - 4

; Storing local variable y = 5
mov  DWORD PTR [rbp-8], 5   ; y is placed at rbp - 8
ARM -mthumb / optimization note
With ARM GCC using -mthumb or -O1 and above, the compiler may omit the frame pointer and access local variables relative to the stack pointer (SP) instead. You may see [sp, #N] in GCC output. AsmWalker samples compile at -O0 (no optimization), so the frame pointer form is used.
Try it in AsmWalker
After executing mov DWORD PTR [rbp-4], 3, check the Stack panel — the cell at rbp-4 now holds the value 3.

7. Stack Overflow

The stack has a size limit. If deep recursion or very large local variables exhaust the stack, a stack overflow occurs and the program crashes.

// Example of excessive recursion (stack overflow immediately)
void infinite(void) {
    int buf[1024];    // allocate 4 KB on stack every call
    infinite();       // never returns → SP decreases without bound
}

Embedded systems like ARM Cortex-M sometimes have only a few KB of stack, so deep recursion and large local arrays must be handled carefully.

Note about AsmWalker
AsmWalker cuts off tracing after 200–500 steps. Infinite recursion produces a "maximum steps exceeded" error rather than a true stack overflow, but it serves as an infinite-loop detector.
