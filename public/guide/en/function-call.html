How Function Calls Work

When you call a function in C, what assembly instructions does the compiler generate? This page walks through all the mechanisms involved in a function call, step by step.

Contents

  2. Passing arguments (calling convention)
  3. Saving the return address
  4. Stack frames
  5. Returning a value
  6. The complete picture

1. Problems a "call" must solve

Consider this C code:

int add(int a, int b) {
    return a + b;
}

int main(void) {
    int result = add(3, 5);
    return 0;
}

When main calls add(3, 5), the CPU must solve three problems:

  1. How to pass the arguments — deliver 3 and 5 to add
  2. Where to return — after add finishes, get back to where main left off
  3. How to receive the return value — use add's result in main

These agreements are called the Calling Convention (ABI — Application Binary Interface). Each architecture has a standard, and compilers follow it strictly.

2. Passing arguments (calling convention)

Arguments are passed via registers rather than on the stack, because register access is much faster than memory.

ARM Cortex-M (AAPCS)

Arguments 1–4 go in r0–r3.
Arguments 5+ go on the stack.

mov r0, #3   ; arg 1: a = 3
mov r1, #5   ; arg 2: b = 5
bl  add

x86-64 (System V AMD64 ABI)

Arguments 1–6 go in rdi rsi rdx rcx r8 r9.
Arguments 7+ go on the stack.

mov edi, 3   ; arg 1: a = 3
mov esi, 5   ; arg 2: b = 5
call add
Why edi instead of rdi?
When an argument is int (32-bit), GCC uses the 32-bit register name (edi) rather than the 64-bit one (rdi). Writing to edi automatically zeroes the upper 32 bits of rdi, so the behavior is identical.

3. Saving the return address

The address of main's next instruction after the call is the return address.

The CPU tracks the current execution position in the PC / RIP. A call or bl instruction overwrites it with the callee's address. Without saving the return address first, there's no way to get back.

ARM: saved in the Link Register (LR / r14)

The BL instruction automatically writes the return address into LR.

bl add      ; LR ← return address
            ; PC ← address of add

; returning from add:
bx lr       ; PC ← LR (back to main)

x86-64: saved on the stack

The call instruction pushes the return address onto the stack.

call add    ; [rsp] ← return address
            ; rsp  -= 8
            ; RIP  ← address of add

; returning from add:
ret         ; RIP ← [rsp] (back to main)
            ; rsp += 8
The fundamental difference between ARM and x86:
ARM saves the return address in one register (LR), so calling another function inside add would overwrite LR. ARM functions therefore begin with push {lr} and end with pop {pc} to save and restore it. x86's call automatically pushes to the stack, so this problem is handled automatically.

4. Stack frames

The region of the stack that a function allocates for its own use — local variables and temporary data — is called its stack frame.

The stack grows from high addresses toward low addresses.

"Releasing" just means moving SP back

When a function returns, its frame is "released" — but this doesn't erase anything in memory.

Both allocation and release are done by moving SP.

Allocate: sub rsp, 16    ; SP decreases → region is "in use"
Release:  add rsp, 16    ; SP increases → region is "available" again
                         ; ※ data remains in memory until the next call overwrites it
"Data persists after release" trap:
If you take a pointer to a local variable and pass it outside the function, the memory it points to is still there briefly after the function returns. But the next function call will overwrite it, causing the pointer to reference garbage. This is the "stack variable address escape" bug common in C.

Building the frame (prologue)

ARM

push {r7, lr}     ; save caller's r7 (FP) and LR
sub  r7, sp, #0   ; r7 (FP) ← current SP
sub  sp, sp, #8   ; allocate space for locals

x86-64

push rbp          ; save caller's rbp
mov  rbp, rsp     ; rbp ← current rsp (frame base)
sub  rsp, 16      ; allocate space for locals

r7 (ARM) and rbp (x86) are the frame pointer (FP / Base Pointer) — they hold the base address of the current frame. Local variables are accessed as "frame pointer ± offset". 

; x86-64: storing local variable a at [rbp-4]
mov DWORD PTR [rbp-4], edi    ; a = first argument (saved to stack)
mov eax, DWORD PTR [rbp-4]   ; eax = a (load)
The frame pointer can be omitted:
With -O1 and above, or ARM Thumb mode (-mthumb), the compiler may skip the frame pointer and use SP-relative addressing for locals instead.

mov DWORD PTR [rsp+4], edi (no frame pointer)

In that case rbp (or r7) is free for other uses. AsmWalker samples use -O0 (no optimization), so the frame-pointer form is generated.
Try it in AsmWalker:
At the push rbp (x86) or push {r7, lr} (ARM) step, the Stack panel shows the frame pointer being pushed. As you step further, the stack grows downward (toward lower addresses).

Tearing down the frame (epilogue)

ARM

pop {r7, pc}   ; restore r7, PC = LR (return)

x86-64

pop rbp        ; restore rbp
ret            ; RIP ← [rsp] (return), rsp += 8
; ※ GCC sometimes uses the leave instruction instead of pop rbp

Stack state during add (x86-64)

High address
┌──────────────────────────────┐
│  ... (main's frame)           │
│  return address (pushed by call) │ ← right after: call add
│  saved rbp                    │ ← right after: push rbp
│  local variable a = 3        │ ← [rbp-4]
│  local variable b = 5        │ ← [rbp-8]
│  ... (free)                   │ ← rsp points here
Low address

  ↑ after pop rbp / ret, rsp moves back up
    and add's frame region becomes "available" (data not erased)

5. Returning a value

The return value is placed in a designated register before returning.

ArchitectureInteger / pointer return value64-bit return value
ARM Cortex-Mr0r0:r1 (high in r1)
x86-64rax (or eax for 32-bit)rax
 
; end of add function (x86-64)
mov edx, DWORD PTR [rbp-4]   ; edx = a
mov eax, DWORD PTR [rbp-8]   ; eax = b
add eax, edx                  ; eax = a + b (function return value)
pop rbp
ret                           ; return to main; result remains in eax

6. The complete picture

The full sequence for main calling add(3, 5) and returning:

1
Place arguments in registers
mov edi, 3 / mov esi, 5 (x86) or mov r0, #3 / mov r1, #5 (ARM)
2
Jump to the function with call / bl
Save the return address to the stack (x86) or LR (ARM), then rewrite PC to point at add.
3
Prologue: build the stack frame
Save the previous frame pointer, set the new frame pointer, allocate space for local variables.
4
Function body executes
Read argument registers, compute the result, store it in eax (x86) or r0 (ARM).
5
Epilogue: tear down the stack frame
Restore the frame pointer, wind the stack back.
6
Return to main with ret / bx lr
Read the return address from the stack (x86) or LR (ARM) and put it back in PC.
7
main uses the return value
The result is in eax (x86) or r0 (ARM).
