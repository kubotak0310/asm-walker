Pointers and Assembly

AsmWalker Guide — What C pointers look like in assembly

Many people learning C hit a wall with pointers and give up. Even when you understand what * and & mean on paper, grasping why they work the way they do can feel elusive.

But once you see the assembly, the picture changes completely. A pointer is nothing more than a register holding an address as a plain integer, and *p becomes a memory-access instruction that reads or writes to that address. That's all there is to it. The moment an abstract pointer is tied to a concrete memory location, understanding clicks into place.

Modern high-level languages like Python and JavaScript deliberately hide memory management in their runtimes. That's a smart design choice for productivity — but it also removes every opportunity to see what a program is actually doing under the hood.

C and C++ are among the very few languages that let you touch memory directly. It's no accident that operating systems, embedded firmware, game engines, and language runtimes are still predominantly written in C/C++. Understanding pointers and memory is a foundation of Computer Science that pays off regardless of which language you work in. We hope this guide is a useful step on that journey.

1. A Pointer Is "a Variable That Holds an Address"

In C, a pointer is a variable that holds the memory address of another variable. From the assembly perspective, a pointer is simply "a register containing an integer value — the address." There is no special syntax; it is stored in the same registers as any other integer. 

// C
int  x = 42;
int* p = &x;   // p holds the address of x

In assembly, p is just a register (or stack slot) that contains an address value.

2. Taking an Address (the & Operator)

C's &x computes "the address where x lives." In assembly, this is done with an address-computation instruction (lea on x86, add/sub on ARM).

ARM — implementing &x

; int x = 42 → placed at [fp-4]
mov  r3, #42
str  r3, [fp, #-4]    ; x = 42

; int *p = &x
sub  r3, fp, #4       ; r3 = fp - 4 (address of x)
str  r3, [fp, #-8]    ; save p = &x on the stack

x86-64 — implementing &x

; int x = 42 → placed at [rbp-4]
mov  DWORD PTR [rbp-4], 42  ; x = 42

; int *p = &x
lea  rax, [rbp-4]           ; rax = rbp - 4 (address of x)
mov  QWORD PTR [rbp-16], rax ; save p = &x on the stack

lea (Load Effective Address) computes an address and stores it in a register without reading any memory — it returns the address value itself. ARM's sub r3, fp, #4 serves the same purpose.

3. Dereferencing (* Operator) and Memory-Access Instructions

C's *p reads or writes the memory at the address held in p — this is called dereferencing. In assembly it appears as a memory-access instruction (ARM: LDR/STR, x86: MOV [...], ...). 

// C
int x = 42;
int* p = &x;
int y = *p;    // read the value at the address p points to → y = 42
*p = 100;      // write 100 to the address p points to → x = 100

ARM — dereferencing

; int y = *p
ldr  r3, [fp, #-8]    ; r3 = p (the address)
ldr  r3, [r3]         ; r3 = *p (the value at that address)
str  r3, [fp, #-12]   ; y = *p

; *p = 100
ldr  r3, [fp, #-8]    ; r3 = p (the address)
mov  r2, #100
str  r2, [r3]         ; *p = 100 (write to that address)

x86-64 — dereferencing

; int y = *p
mov  rax, QWORD PTR [rbp-16]  ; rax = p (the address)
mov  eax, DWORD PTR [rax]     ; eax = *p (the value at that address)
mov  DWORD PTR [rbp-20], eax  ; y = *p

; *p = 100
mov  rax, QWORD PTR [rbp-16]  ; rax = p (the address)
mov  DWORD PTR [rax], 100     ; *p = 100 (write to that address)
Notice the two memory accesses
Dereferencing always involves two steps: "1st: read the pointer variable to get the address" and "2nd: read or write the data at that address." This double indirection is one reason pointer-heavy code can be slower than direct variable access.
Try it in AsmWalker
At a step like ldr r3, [r3] or str r2, [r3] (x86: mov eax, [rax]), the base register inside [...] in the Register panel shows a purple "ptr" badge. This badge indicates that the register is being used as a pointer at this specific instruction. Also check the Stack panel to see the value at the corresponding address change.

4. Passing Pointers to Functions

When you pass a pointer as a function argument in C, assembly generates an instruction to copy the address value into an argument register. Inside the function, reading or writing through that address directly modifies the caller's variable. This is the true nature of "pass by reference." 

// C
void double_it(int* p) {
    *p = *p * 2;   // modify the caller's variable
}

int main(void) {
    int x = 5;
    double_it(&x);  // pass address of x → x becomes 10
}

ARM — caller side

; calling double_it(&x)
sub  r0, fp, #8    ; r0 = &x (address as arg 1)
bl   double_it     ; function call

x86-64 — caller side

; calling double_it(&x)
lea  rdi, [rbp-4]  ; rdi = &x (address as 1st arg)
call double_it     ; function call

ARM — callee (double_it)

; r0 = &x was passed in
ldr  r3, [r0]      ; r3 = *p (read x's value)
mov  r2, r3, lsl #1 ; r2 = r3 * 2
str  r2, [r0]      ; *p = r2 (overwrite x)

x86-64 — callee (double_it)

; rdi = &x was passed in
mov  eax, DWORD PTR [rdi]   ; eax = *p
add  eax, eax               ; eax *= 2
mov  DWORD PTR [rdi], eax   ; *p = eax (overwrite x)
Try it in AsmWalker
Confirm that the value of x in the Stack panel changes before and after the function call. Just before bl double_it / call double_it, r0/rdi shows a blue "Arg 1" badge. Inside the function, at ldr r3, [r0] / mov eax, [rdi], r0/rdi shows a purple "ptr" badge, confirming pointer dereferencing.

5. Arrays and Pointers

In C, an array name behaves like a pointer to its first element. In assembly, you can see the explicit calculation: "base address of array + index × element size." 

// C
int arr[3] = {10, 20, 30};
int v = arr[2];   // read 2 * 4 = 8 bytes past the start

ARM — array access

; get address of arr[0]
sub  r3, fp, #20   ; r3 = &arr[0]

; access arr[2] (offset = 2 * 4 = 8)
ldr  r3, [r3, #8]  ; r3 = arr[2]

x86-64 — array access

; get address of arr[0]
lea  rax, [rbp-16]          ; rax = &arr[0]

; access arr[2] (offset = 2 * 4 = 8)
mov  eax, DWORD PTR [rax+8] ; eax = arr[2]

When iterating an array with a loop, the index calculation (index * 4) may appear as a shift instruction (lsl #2) or as SIB addressing in x86. 

; x86-64: SIB addressing for arr[i]
mov  eax, DWORD PTR [rax + rcx*4]
;                         ↑ CPU computes index × sizeof(int) automatically
Try it in AsmWalker
Array-element access instructions show a green "arr" badge. The Stack panel also highlights each array cell in green so you can visually track which element is being accessed.

6. Pointer Arithmetic

Adding an integer to a pointer advances the address by that many elements (not bytes). This is pointer arithmetic. In assembly, the element-size multiplication is visible directly. 

// C
int arr[5] = {1, 2, 3, 4, 5};
int* p = arr;      // p → arr[0]
p = p + 2;         // p → arr[2] (address advances by 2*4 = 8)
int v = *p;        // v = 3

ARM — pointer arithmetic

; p = p + 2  (int* so +8 bytes)
ldr  r3, [fp, #-8]    ; r3 = p
add  r3, r3, #8       ; r3 += 2 * sizeof(int)
str  r3, [fp, #-8]    ; p = r3

x86-64 — pointer arithmetic

; p = p + 2  (int* so +8 bytes)
mov  rax, QWORD PTR [rbp-16]  ; rax = p
add  rax, 8                   ; rax += 2 * sizeof(int)
mov  QWORD PTR [rbp-16], rax  ; p = rax
Pointer arithmetic is in elements, not bytes
In C, p + 1 means "address + sizeof(*p)", not "address + 1". At the assembly level everything is always in bytes, so C's p + 2 becomes +8 for an int* but only +2 for a char*.

7. Double Pointers

A double pointer (int**) is "a pointer to a pointer" — an address of an address. In assembly it appears as three levels of memory access. 

// C
int  x  = 42;
int* p  = &x;
int** pp = &p;

int v = **pp;   // follow pp → p → x

x86-64 — double dereference

; v = **pp
mov  rax, QWORD PTR [rbp-32]   ; rax = pp (address of address)
mov  rax, QWORD PTR [rax]      ; rax = *pp = p (the address)
mov  eax, DWORD PTR [rax]      ; eax = **pp = x (the value)

Every additional level of indirection adds one more memory access. In real C programs, double pointers appear frequently: char** argv for argument arrays, Node** for linked list updates, and so on.

8. Summary

C operationAssembly equivalent
&x (take address)lea rax, [rbp-4] / sub r3, fp, #4 — compute address
*p (read)mov eax, [rax] / ldr r3, [r3] — memory load
*p = v (write)mov [rax], eax / str r3, [r3] — memory store
p + n (pointer arithmetic)add rax, n * sizeof(*p) — byte-level addition
arr[i] (array access)[base + i * size] — base + offset calculation
**pp (double dereference)Three memory accesses: pp → p → value

Seen through assembly, a pointer is nothing more than "a register holding an address." Understanding that every dereference costs an extra memory access makes both the performance implications and the common bugs far clearer.
