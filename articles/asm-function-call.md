## 関数呼び出しは「魔法」じゃなかった

C言語を書くとき、関数呼び出しは当たり前に使う。

```c
int result = add(3, 5);
```

でも「裏で何が起きているか」を説明できる人は意外と少ない。引数はどこに渡るのか。戻り値はどこから来るのか。呼び出し元に「戻る」とはどういうことか。

これを全部、アセンブラは正直に書いている。

---

## 引数はレジスタに入れて渡される

```c
int add(int a, int b) {
    return a + b;
}

int main(void) {
    int x = add(3, 5);
}
```

`add(3, 5)` を呼び出すとき、ARMのアセンブラはこうなる。

```
mov  r0, #3     ; 第1引数を r0 に
mov  r1, #5     ; 第2引数を r1 に
bl   add        ; add を呼び出す
```

`mov r0, #3` ——3という値をレジスタ r0 に入れる。それだけ。

**引数は関数に「渡す」のではなく、あらかじめ決まったレジスタに「置いておく」**。関数側がそこから読みに来る。ARM では r0〜r3 が引数レジスタだ。

## 引数が5個以上になるとスタックを使う

ARM の ABI では **r0〜r3 の4つが引数レジスタ**として予約されている。5個目以降の引数はレジスタに入れる場所がないため、スタック経由で渡される。

```
mov  r0, #1      ; 第1引数 → レジスタ（高速）
mov  r1, #2      ; 第2引数 → レジスタ（高速）
mov  r2, #3      ; 第3引数 → レジスタ（高速）
mov  r3, #4      ; 第4引数 → レジスタ（高速）
push #5          ; 第5引数 → スタック（メモリアクセス）
bl   func
```

レジスタは CPU 内部にあるので読み書きが**瞬時**に終わる。スタックはメモリ上にあるのでアクセスに時間がかかる。

これが**組み込みプログラマが引数を4つ以内に抑えることにこだわる理由**だ。「なんとなくそう習った」ではなく、アセンブラを見れば一目瞭然の、れっきとした性能上の理由がある。設計の段階で引数が5個を超えそうなら、構造体にまとめてポインタで渡す——そういう判断もここから来ている。

## フレームポインタはローカル変数の「基準点」

`push {r7, lr}` の r7 は**フレームポインタ（FP）**だ。

関数に入ると、まず SP を下げてローカル変数の領域を確保する。

```
push {r7, lr}       ; r7 と lr を保存
sub  sp, sp, #8     ; ローカル変数用に 8 バイト確保
add  r7, sp, #0     ; r7 = 現在の SP（フレームの基準点を記録）
```

r7 を基準にすると、ローカル変数のアドレスが**固定オフセット**で表せる。

```
str  r0, [r7, #4]   ; 変数 a は r7+4 の場所
str  r1, [r7, #0]   ; 変数 b は r7+0 の場所
```

SP は PUSH/POP のたびに動くが、**r7 は関数の間ずっと動かない**。だから `[r7, #4]` は関数内のどこからでも安定して変数 a を指せる。

```
高アドレス
  [ 保存した LR  ]
  [ 保存した r7  ]
  [ 変数 a      ] ← [r7, #4]
  [ 変数 b      ] ← [r7, #0] ← r7（FP）はここを指す
低アドレス          ← SP はここ
```

なお `-O1` 以上の最適化をかけると、GCC はフレームポインタを省略して SP 相対アクセス（`[sp, #4]`）だけで済ませることが多い。フレームポインタは主に `-O0`（デバッグビルド）での産物だ。

**アセンブラを学ぶ段階では `-O0`（最適化なし）で読むのがベター**だ。最適化がかかると命令が省略・並び替えされ、C コードとアセンブラの対応が追いにくくなる。「まず動きを理解する」には最適化なしが正直な出力をしてくれる。AsmWalker のサンプルもデフォルトで `-O0` を使っているのはそのためだ。

## 「bl」命令が関数呼び出しの正体

`bl add` ——この1命令で関数呼び出しが起きる。

`bl` は **Branch with Link** の略で、2つのことを同時にやる。

1. **PC（プログラムカウンタ）を `add` のアドレスに書き換える**（ジャンプ）
2. **LR（リンクレジスタ）に「次の命令のアドレス」を保存する**（戻り先を記憶）

LR に保存するのが重要だ。これが「関数が終わったら戻ってくる」仕組みの核心だ。

---

## 戻り先アドレスはレジスタに入っている

呼び出された `add` 関数の中はこうなる。

```
add:
  push {r7, lr}       ; フレームポインタと LR を保存
  sub  sp, sp, #8     ; スタックを確保
  ...（計算処理）
  add  r0, r2, r3     ; 結果を r0 に入れる（戻り値）
  pop  {r7, pc}       ; スタックから LR を取り出し PC に入れる
```

最後の `pop {r7, pc}` ——LR（戻り先アドレス）をスタックから取り出して、PC に直接書き込む。

PCが書き変わる → 次に実行する命令が変わる → 呼び出し元に戻る。

**「関数から戻る」とは、LR に保存しておいた番地に PC を書き換えることだった。**

---

## 戻り値は r0 に入れて返す

`add` 関数は `a + b` の結果を r0 に入れて終わる。

```
add  r0, r2, r3    ; r0 = a + b
```

呼び出し元は `bl add` の直後に r0 を読む。それが戻り値だ。

```
; main に戻ってきた後
str  r0, [r7, #4]  ; r0（戻り値）を変数 x に書く
```

**戻り値は「返す」のではなく、r0 に「置いておく」だけ**。呼び出し元が読みに来る。

## スタックフレームとは何か

関数が呼ばれるたびに、スタック上に「その関数専用の作業領域」が作られる。これがスタックフレームだ。

```
高アドレス
  [ main のフレーム  ]
  [ add のフレーム   ] ← add が動いている間だけ存在
低アドレス
```

`add` が終わると、スタックポインタが元に戻り、`add` のフレームは消える。

## まとめ：関数呼び出しの全貌

| やること | アセンブラでの実装 |
|---|---|
| 引数を渡す | r0〜r3 にあらかじめ値を入れる |
| 関数を呼ぶ | `bl` でジャンプ + LR に戻り先を保存 |
| 戻り値を返す | r0 に結果を入れて終わる |
| 呼び出し元に戻る | LR の値を PC に書き込む |

関数呼び出しは魔法ではない。**レジスタとスタックを使った、決まり事の積み重ね**だ。

その決まり事を「ABI（Application Binary Interface）」と呼ぶ。ARMのABIを理解すると、デバッガのスタックトレースが読めるようになり、クラッシュの原因が追えるようになる。
